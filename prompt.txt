Tu travailles dans le dépôt preview-experiments / preview-operator.

Objectif :
Préparer le dépôt pour une étude empirique TSE-ready sur checkpoint-based database state isolation dans des Kubernetes preview environments, en conservant RQ5 sur l’idempotence/convergence de l’opérateur après redémarrage.

IMPORTANT :
- Ne lance AUCUNE expérience.
- Ne fabrique AUCUNE donnée.
- Ne supprime AUCUN CSV.
- Ne modifie pas la logique de checkpoint/restore sauf si explicitement demandé.
- Ne modifie pas la logique métier des applications sujet.
- Les changements doivent être minimaux, réversibles et documentés.
- Commence par explorer le dépôt et produire un plan détaillé AVANT toute modification.
- Attends validation avant d’appliquer les changements.
- Tout chiffre du papier doit être régénérable depuis des CSV figés, jamais depuis un tracker live.

1. Explorer le dépôt.
2. Identifier les dossiers de résultats existants.
3. Identifier les CSV final / obsolete / diagnostic / partial.
4. Identifier les scripts d’analyse existants.

6. Identifier où sont générés les résultats suite-level.
7. Identifier le code RQ5 / idempotence :
   - kill steps disponibles
   - méthode de redémarrage de l’opérateur
   - métriques actuellement enregistrées
   - risques de concurrence avec les autres expériences
8. Produire un plan de modification AVANT toute modification avec :
   - fichiers à modifier
   - fichiers à créer
   - risques
   - critères d’acceptation
   - ordre recommandé des commits

PHASE 4 — Corrections harness S2/S4/S5
Localiser les assertions défectueuses connues :
- S2 : baseline codée en dur incompatible avec état post-install/post-seed.
- S4 : endpoint nécessitant autorisation ou query params absents.
- S5 : payload invalide / statut HTTP attendu incorrect / smoke qui échoue dès l’état migré.

Pour chaque sujet :
- expliquer le symptôme
- expliquer la cause racine
- proposer une correction minimale
- appliquer la correction uniquement si elle est clairement justifiée
- documenter dans HARNESS_FIXES.md :
  - symptôme
  - cause racine
  - correction appliquée
  - pourquoi la correction ne masque pas un échec d’isolation
  - impact attendu
  - fichiers modifiés

Contraintes :
- Ne pas supprimer une assertion sans justification.
- Préférer les baselines runtime aux constantes codées en dur.
- Ne pas transformer un vrai bug applicatif en succès artificiel.
- Si une correction est ambiguë, la signaler et ne pas deviner.

Analyses obligatoires :

RQ1 Reliability :
- failure rate par sujet / suite / condition
- Wilson ou Clopper-Pearson confidence intervals
- risk difference en percentage points
- Fisher exact test
- odds ratio avec correction Haldane si nécessaire
- effect label
- séparation suite-level vs assertion-level

Assertion-level decomposition :
- nombre d’assertions par catégorie
- taux d’échec par catégorie
- isolation-sensitive assertions
- baseline-invalid assertions
- functional assertions
- infra-related failures
- table “suite-level failure vs root-cause category”

DB-state verification :
- row-count match par table
- content-hash match par table
- snapshot_hash_global match
- dirty_state_detected
- restore_verified
- table par sujet et par restore step

RQ2 Concurrency :
- failure rate par sujet / K / condition
- observed_previews vs declared_K
- completion_rate
- exclusion ou flag des batches incomplets
- Fisher exact test si données complètes
- commentaire automatique si K=8 incomplet

RQ3 Performance :
- mean, median, SD, IQR, p95
- checkpoint_total
- save_time
- restore_time
- pipeline_duration
- Mann-Whitney U
- Vargha-Delaney A12
- Cliff’s delta
- comparaison no isolation vs checkpoint_restore
- si migration_reset existe : comparaison mesurée
- sinon : marquer migration_reset comme theoretical/derived, pas measured

RQ4 Mutation :
- mutation score par seed condition
- killed / survived / invalid mutants
- confidence intervals
- pairwise McNemar
- gestion explicite du cas n01=n10=0
- matrice de concordance mutants × seed condition
- minimum detectable effect / power analysis
- ne pas interpréter les sujets dont les mutants ne sont pas alignés avec le SUT

RQ5 Idempotence and convergence :
- lire uniquement les CSV RQ5 figés
- calculer par sujet et kill_step :
  - total restarts
  - succeeded restarts
  - failed restarts
  - convergence_time_sec mean / median / p95
  - operator_unavailable_sec
  - webhook_unavailable_sec si disponible
  - duplicate_job_count
  - lost_status_count
  - orphaned_resource_count
  - final_state_consistent
  - pipeline_succeeded_after_restart
- produire une table :
  subject_id, kill_step, repetitions, success_rate, median_convergence_time, p95_convergence_time, duplicate_job_count, lost_status_count, verdict
- si les métriques nécessaires sont absentes, produire un warning explicite :
  “RQ5 cannot support confirmatory claims because convergence metrics are missing.”
- ne jamais mélanger RQ5 avec les runs RQ1/RQ2/RQ3.

PHASE 7 — Protection RQ5 obligatoire
Conserver RQ5, mais ajouter une garde empêchant toute exécution parallèle.

Implémenter un mécanisme de lock global, par exemple :
.experiment_lock
ou un fichier équivalent dans le dossier runtime/logs.

Règles :
- Si RQ5 démarre alors qu’une autre expérience tourne : fail fast.
- Si une autre expérience démarre pendant RQ5 : fail fast.
- Le message d’erreur doit expliquer :
  “RQ5 restarts the preview operator and cannot run concurrently with other experiments.”
- Documenter cette contrainte dans README et REPRODUCE.md.
- Ajouter une option explicite :
  --force-rq5-alone
  ou équivalent si le harness utilise des arguments CLI.
- Ne pas lancer RQ5 automatiquement dans une chaîne parallèle.
