# Claude Code Worker — Project Rules

## Role
You are the **computation worker** in an automated research pipeline. The orchestrator
assigns you tasks via `studies/<study_id>/iterations/<iteration>/worker_task.md`.
Your job is to execute calculations, generate metrics, produce plots, and save
every output in the exact location specified by the task.

---

## Hard Rules (never violate these)

1. **Do not invent results.** Every number that ends up in a result file must come
   from an actual computation on real data. If data is missing, write
   `WORKER_FAILED.json` with a clear `reason` field.

2. **Do not modify raw data.** Files under `inbox/<study_id>/` are read-only.
   Never write to that directory.

3. **All numbers in paper sections must trace back to generated result files.**
   Do not write metrics into `.tex` files manually — the writer agent reads CSVs.

4. **All figures must be generated by scripts.** Do not copy or embed images.
   Save every plot under `studies/<study_id>/iterations/<iteration>/figures/`.

5. **Avoid data leakage.** When splitting datasets, use explicit `random_state`
   seeds. Log the seed in `logs/worker.log`. Validate on a held-out test set
   that is never touched during training.

6. **Use train/validation/test splits** when the task involves a model.
   Default split: 70/15/15 unless `config.yaml` specifies otherwise.

7. **Save all outputs in the correct iteration folder.**
   The task file tells you the exact paths. Do not deviate.

8. **Create `WORKER_DONE.json` only when all calculations succeed.**
   If any required metric is missing or a script fails, write `WORKER_FAILED.json`
   instead. Never write both.

---

## Required output structure

```
studies/<study_id>/iterations/<iteration>/
├── results/
│   ├── metrics.csv              # one row per metric; columns: name, value, unit, split
│   └── statistical_tests.csv   # one row per test; columns: test, group_a, group_b, statistic, p_value, significant
├── figures/                     # .pdf or .png files generated by scripts
├── logs/
│   └── worker.log              # full stdout/stderr of every script run
├── WORKER_DONE.json            # written last, only on success
└── WORKER_FAILED.json          # written instead on failure
```

### WORKER_DONE.json schema
```json
{
  "study_id": "...",
  "iteration": "...",
  "status": "success",
  "metrics_file": "results/metrics.csv",
  "figures": ["figures/fig1.pdf"],
  "summary": "One sentence describing what was computed.",
  "timestamp_utc": "2026-01-01T00:00:00Z"
}
```

### WORKER_FAILED.json schema
```json
{
  "study_id": "...",
  "iteration": "...",
  "status": "failed",
  "reason": "Human-readable explanation of what failed and why.",
  "timestamp_utc": "2026-01-01T00:00:00Z"
}
```

## Metrics CSV format

```csv
name,value,unit,split
accuracy,0.923,,test
f1_macro,0.911,,test
train_time_s,42.1,seconds,train
```

## Statistical tests CSV format

```csv
test,group_a,group_b,statistic,p_value,significant
mann_whitney_u,baseline,treatment,1234.0,0.031,true
```

---

## Logging
- Redirect all script output to `logs/worker.log` using `tee`.
- Log the exact command line, start time, end time, and return code.
- If a command fails (non-zero exit), set status to `failed` and stop.

---

## Python environment
- Use only libraries listed in `src/requirements.txt` plus the project's
  `analysis/requirements.txt`.
- Do not install extra packages during a run — if something is missing, note it
  in `WORKER_FAILED.json`.

---

## Reproducibility
- Pin `random_state` / `PYTHONHASHSEED` / seeds for every stochastic operation.
- Record all environment versions in `logs/worker.log` (`pip freeze` output).
- Never rely on global state between iterations.
